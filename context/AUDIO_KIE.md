# 🎙 Аудиогенерация через KIE.ai (модели ElevenLabs и Suno) — Контекст функции

## Обзор
Данная функциональность реализует поддержку генерации аудио через провайдера KIE.ai для моделей ElevenLabs и Suno. Поддерживаются разные типы аудио-генерации:
- преобразование текста в речь,
- расшифровка речи в текст,
- изоляция аудио,
- создание звуковых эффектов,
- синтез диалогов,
- генерация музыки и аудио с помощью моделей Suno,
- поддержка расширенных параметров Suno (customMode, instrumental, стиль, длительность и др.).

Функция интегрирована в существующую архитектуру backend, включая проверку баланса токенов пользователя, обработку асинхронных задач с поллингом, fallback между провайдерами и уведомления о статусах в реальном времени через WebSocket.

## Техническая архитектура

### Основные модули:

- **AI Providers**
  - Класс `KieProvider` — реализует методы генерации аудио для моделей ElevenLabs и Suno, формирует корректные запросы, отправляет в API KIE.ai, обрабатывает ответы и статусы задач.
  - Реализован метод `generateAudio` с ветвлением на поддержку ElevenLabs и Suno, а также методы для генерации и статуса лирики (`generateLyrics`, `getLyricsTaskStatus`).

- **Модуль генерации**
  - `GenerationService` — контролирует проверку баланса, создание заявки, постановку в очередь.
  - `GenerationConsumer` — обрабатывает задачи, вызывает `AiProvidersService.generateAudio`, опрашивает статус и сохраняет результаты.
  - `AiProvidersService` — реализует fallback между провайдерами, вызывает соответствующие методы провайдера, включая KieProvider с поддержкой Suno и ElevenLabs.

---

## Поток данных

1. Клиент отправляет запрос на `/api/v1/generation/audio` с телом соответствующим `AudioGenerationDto`, включая выбор модели Suno или ElevenLabs.
2. Сервис проверяет баланс токенов пользователя и резервирует стоимость использования.
3. Создаётся запись о запросе генерации с статусом `pending` в базе.
4. Задача помещается в очередь на обработку.
5. Consumer забирает задачу из очереди и вызывает генерацию через `AiProvidersService`.
6. Провайдер KIE.ai формирует запрос в зависимости от модели и отправляет в API, получает `taskId` асинхронной задачи.
7. Consumer запускает опрос статуса задачи с помощью методов провайдера.
8. Статус и прогресс сохраняются в базе, обновления отправляются пользователю через WebSocket.
9. По завершении результата - сохраняется, происходит списание токенов/обработка ошибок, уведомления.
